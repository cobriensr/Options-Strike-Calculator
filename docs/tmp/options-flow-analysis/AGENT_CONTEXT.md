# Lottery Finder — Agent Bootstrap

If you're a fresh Claude session being asked to "extend the Lottery
Finder analysis with a new day" or "re-run the lottery analysis,"
this file is your ground truth. Read it top to bottom before doing
anything.

## What this project is (1 paragraph)

The Lottery Finder is a **signal detector** (NOT a backtested
profitable strategy) that surfaces option-flow setups historically
associated with explosive right-tail outcomes (≥+200% EoD peak in the
training window). It runs on the per-tick OPRA stream — a v4
event-based trigger fires every time vol/OI re-accelerates with high
ask-side fraction on a near-the-money option. Each fire is enriched
with a RE-LOAD discriminator (continuation pattern: same chain,
bigger burst, lower entry) and a cheap-call-PM selection rule (call +
PM session + entry < $1) that was the strongest discriminator in the
15-day backtest. **Edge concentrated in 1-2 outlier days per 15** —
the user trades it as informational signal, not as an autonomous
system.

## The production rule + thresholds

Codified in `api/_lib/lottery-finder.ts`:

```ts
LOTTERY_SPEC_V4 = {
  volToOiWindowMin: 0.05,   // 5% of OI in the rolling 5-min window
  volToOiCumMin: 0.10,      // chain context: 10% of OI total today
  ivMin: 0.35,              // window mean IV
  absDeltaMin: 0.13,        // |window mean delta|
  askPctMin: 0.52,          // ask-side fraction
  dteMax: 7,                // max DTE eligible
  cntWindowMin: 5,          // min prints in the window
  cooldownMin: 5,           // min minutes between fires on same chain
};
isCheapCallPm = (type === 'C' && tod === 'PM' && entry < $1)
isReload = (burstRatio ≥ 2 AND entryDropPct ≤ -30)
```

Universe: V3 list (Mode A 0DTE intraday) ∪ EXTENDED list (Mode B DTE 1-3
trend). See `LOTTERY_V3_TICKERS` + `LOTTERY_EXTENDED_TICKERS` in the
same file. Mode B requires `|strike/spot - 1| ≤ 0.10` (in-play gate).

## Honest caveats (do NOT skip)

- **Most days lose.** Win rate is high on lottery-tagged trades but
  the dollar-realized profile is fat-tailed: a few +500% peaks carry
  the EV, hundreds of -50–-90% losses drag it.
- **Edge is regime-dependent.** The 15-day seed window had +$672 in
  cherry-picked top-3/day cumulative $ realized; **88% of that came
  from 2026-04-21 alone**. Without that day, only `act30_trail10` is
  barely positive.
- **Macro features did NOT improve the rule** in the 15-day backtest.
  Every macro-augmented AND-rule (p30/p31) UNDERPERFORMED the
  cheap-call-PM-only baseline on total realized $. Macro is shipped
  as display-only badges, never as filter chips. See spec
  Appendix A for the full negative-findings table.
- **Sample size is small.** 15 trading days, ~50 cheap-call-PM
  RE-LOAD fires. Stress test (p29) shows the result is sensitive to
  outlier days; we re-validate with each new day appended.

## What NOT to do

- **Do NOT change the production rule silently.** If a new day's
  analysis fails the pass criteria in PIPELINE.md Step 6, open a new
  spec doc explaining the regime change before touching code.
- **Do NOT add macro features as selection signals.** They've been
  tested twice (p30 + p31, see Appendix A). Adding them as filters
  would degrade the realized $ profile based on the 15-day data.
- **Do NOT rerun production code against historical parquet without
  a flag.** The cron reads `ws_option_trades`; the parquet is a
  separate source for the offline analysis. Crossing the streams
  would write duplicates with different timestamps.
- **Do NOT skip Step 9 of PIPELINE.md.** Append the entry to
  PIPELINE_STATE.md so the next agent (or future-you) can see the
  validation history without re-running the stack.
