# docs/tmp/ — Research Scratch Space

Ephemeral EDA outputs, ad-hoc analyses, and intermediate research artifacts. **Not a stable reference.** Findings here may be refuted, superseded, or replaced by later work.

## Convention

- New files in this folder are **gitignored by default** (see [.gitignore](../../.gitignore) line ~191).
- A file becomes tracked only when committed with `git add -f docs/tmp/<path>`. The 147 currently-tracked files are intentional captures from research worth referencing in commits or follow-up specs.
- When a finding here becomes load-bearing for trade decisions or future work, **promote it out**:
  - Operational guide → root `docs/*.md` runbook
  - Design context → `docs/superpowers/specs/{topic}-{date}.md`

## What you'll find

Mixed by file extension:

| Type   | Source                                                    | Notes                                |
| ------ | --------------------------------------------------------- | ------------------------------------ |
| `.md`  | Findings reports, EDA narratives, feature audits          | Dated; usually one-off               |
| `.py`  | Single-purpose analysis scripts                           | Pair with a `.md` summary nearby     |
| `.mjs` | Node ad-hoc queries (Neon, UW API)                        | Usually `.env.local` aware           |
| `.csv` | Snapshots of intermediate result sets                     | Generated by adjacent `.py` / `.mjs` |
| `.png` | Plots, screenshots from manual capture or matplotlib runs | Not the ML pipeline output (that's in `ml/plots/` and Vercel Blob) |

## When to read what's here

- **You're investigating a current bug or proposed change** → grep here for prior research on the same topic. Verify any conclusion against current code before acting; this folder ages fast.
- **You're writing a new spec** → check if there's existing research that informs your scope. A few of these have already shaped specs in [`../superpowers/specs/`](../superpowers/specs/).
- **You're trying to understand a recent commit** → the commit message will usually name the relevant tmp/ artifact.

## When NOT to read what's here

- As a tutorial or onboarding doc — see [docs/INDEX.md](../INDEX.md) for those.
- For current-state architecture — see [docs/ARCHITECTURE.md](../ARCHITECTURE.md). Code in `api/` is the source of truth; tmp/ markdowns are frozen-in-time analyses.

## Promoting a finding

When something in tmp/ becomes durable:

1. Write a new spec under `docs/superpowers/specs/` summarizing the finding with proper Status frontmatter.
2. Reference the tmp/ artifact from the spec with its full path so the provenance is preserved.
3. Leave the tmp/ file in place — it's the audit trail. Don't delete past research; the value comes from being able to revisit the raw output.

If a finding turns out to be wrong, **don't delete it either**. Add a short footer note (`Status: refuted by <new finding>`) so the next person investigating the same question doesn't repeat the work.
