# Cloudflare R2 archive for UW Full Tape parquets

**Date:** 2026-05-10
**Status:** in progress (waiting on R2 bucket creation)

## Goal

Move the UW Full Tape parquet archive (currently ~99 GiB, growing
~1.2 GiB/day) off the laptop's local SSD and onto Cloudflare R2 for
durability + cross-machine access. Keep local copies as a fast cache
for active development; treat R2 as the source of truth.

## Why R2 specifically

R2's pricing is asymmetric vs. competitors in exactly the right way:

| Provider          | Storage $/GB/mo | Egress $/GB |
| ----------------- | --------------- | ----------- |
| **Cloudflare R2** | **$0.015**      | **$0**      |
| Vercel Blob       | $0.030          | $0.030      |
| AWS S3 Standard   | $0.023          | $0.090      |
| Backblaze B2      | $0.005          | $0.010      |
| AWS Glacier IR    | $0.004          | $0.030 retr |

For an archive that the ML pipeline + ad-hoc backtests will read
repeatedly, R2's $0 egress dominates. 100 GiB stored for $1.50/mo and
unlimited reads is the right fit.

## Setup walkthrough — Cloudflare side (one-time, ~5 minutes)

1. **Sign up / log in** at https://dash.cloudflare.com (free account
   is fine; R2 enables on a separate billing flow).
2. **Enable R2** — left nav → **R2 Object Storage** → "Get Started".
   Adds a payment method on file. Free tier covers 10 GiB storage +
   1M Class A ops + 10M Class B ops per month; the existing 99 GiB
   archive takes us out of the free tier.
3. **Create bucket** — name it `theta-options-fulltape` (or whatever;
   that name goes in `R2_BUCKET_FULLTAPE`).
4. **Generate API token** — R2 page → **Manage R2 API Tokens** →
   **Create API Token**:
   - Permissions: **Object Read & Write**
   - Specify bucket: the one you just made
   - TTL: Forever (or rotate manually later)
5. **Save these values** (you won't see the secret again):
   - Access Key ID
   - Secret Access Key
   - Endpoint URL — looks like `https://<account-id>.r2.cloudflarestorage.com`
   - Bucket name

## Env vars to add to `.env.local`

```bash
# ── Cloudflare R2 — UW Full Tape parquet archive ──
R2_ACCOUNT_ID=               # visible in the dashboard URL
R2_ACCESS_KEY_ID=            # from step 4 above
R2_SECRET_ACCESS_KEY=        # paired with the access key
R2_ENDPOINT_URL=             # https://<account-id>.r2.cloudflarestorage.com
R2_BUCKET_FULLTAPE=          # e.g. theta-options-fulltape
```

Do NOT commit these values. The `.env.example` should grow the same
keys (without values) — this spec is the placeholder until the
`.env*` edit-guard is loosened or someone adds them by hand.

## Files created

- `scripts/upload-fulltape-to-r2.py` — one-shot + repeatable upload
  with idempotent skip. Lists current R2 keys, only uploads files
  missing from the bucket. `--force YYYY-MM-DD` re-uploads a specific
  date; `--dry-run` lists changes without transferring.
- `ml/src/utils/r2.py` — read helper:
  - `read_fulltape(date)` — pandas DataFrame, local cache hit first,
    falls back to R2.
  - `query_fulltape(sql, dates)` — DuckDB-against-R2 query that only
    downloads the column ranges DuckDB needs. The right tool for
    aggregations / GROUP BY work that doesn't need the full file.

## Workflow

**One-shot seed** (after env vars are set):

```bash
ml/.venv/bin/python scripts/upload-fulltape-to-r2.py
```

Will discover the local 99 GiB archive in
`~/Desktop/Eod-Full-Tape-parquet`, list R2's contents, and upload the
delta. First run = ~99 GiB transferred. Cloudflare ingress is free,
so this costs nothing beyond storage.

**Daily incremental** (call after `make ingest-fulltape`):

```bash
ml/.venv/bin/python scripts/upload-fulltape-to-r2.py
```

Idempotent — only the new day uploads. Add this to a Makefile target
or a post-ingest hook later.

**Reading from ML pipeline:**

```python
from utils.r2 import read_fulltape, query_fulltape

# Pandas DataFrame (uses local cache if present)
df = read_fulltape("2026-03-15", columns=["ticker", "strike", "size"])

# DuckDB aggregate over many days, R2-resident
agg = query_fulltape(
    "SELECT ticker, SUM(size) AS total_size FROM __TAPE__ GROUP BY ticker",
    dates=["2026-03-15", "2026-03-16", "2026-03-17"],
)
```

`query_fulltape` is the killer feature: a 6-month aggregate that
would historically require downloading 100+ GiB of parquet now runs
against R2 in seconds and only pulls the few MB DuckDB actually
needs (parquet row-group metadata + ranged column reads).

## Local cache eviction strategy

The local `~/Desktop/Eod-Full-Tape-parquet` directory keeps growing
even after R2 has the data. Once R2 is the source of truth, the
local copy is just a cache. Possible eviction policies (decide
later):

- **Manual**: delete `*-fulltape.parquet` for any day older than X.
- **LRU by mtime**: cron that deletes files with `atime > 30d`.
- **Keep last N days**: simpler — `ls | sort | head -N | xargs rm`.

For now the local archive stays as-is. Eviction is a future cleanup
when local disk pressure returns.

## Open questions

- **Bucket region**: Cloudflare lets you pin a region (Wnam, ENam,
  Wnam, etc.) or auto-distribute. For 1-user workload, `auto` is
  fine. Pinning may save ~50ms of latency from US Central if it
  matters. Default: `auto`.
- **Public access**: bucket should be private (UW data is paid for,
  not for redistribution). Don't enable public bucket. Don't enable
  R2 → custom domain unless adding a backend that serves it
  authenticated.
- **Lifecycle policies**: R2 supports object lifecycle rules
  (e.g., archive after 90 days). Not needed for this archive — every
  day stays "hot" forever in this use case. Skip.

## Risks

- **Token leak**: the access key + secret give read+write on the
  bucket. Treat like a database credential — never commit, rotate if
  exposed (R2 dashboard → revoke + regenerate).
- **Bandwidth on first upload**: 99 GiB at ~50 Mbps upload = ~5
  hours. May want to start it overnight. Cloudflare ingress is free
  so the only cost is wall-clock time.
- **Vercel functions can't reach R2 for queries**: Vercel functions
  do support outbound HTTPS, so `pd.read_parquet('s3://...')` would
  work in principle, but pulling 100s of MB into a Vercel function
  is a bad fit. Read access from the app stays as today (Postgres
  for live queries); R2 reads happen from ML pipeline + local
  research.
