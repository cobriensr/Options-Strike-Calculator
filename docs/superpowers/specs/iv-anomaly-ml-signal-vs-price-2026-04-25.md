# IV Anomaly ML Study — Signal vs Price Movement

## Goal

Use the labeled `iv_anomalies` dataset (15,886 backfill rows + ongoing
live rows) to answer two questions in order:

1. **(Backtest)** Per-ticker, per-DTE — which exit playbook (sell on ITM
   touch / sell at peak / hold to EOD) wins? What's the win rate, mean
   PnL, max loss?
2. **(Stratification)** Conditioned on backtest outcomes — which gates
   and signal combinations are producing the actionable alerts vs the
   noise? Per-ticker low win rates inform whether that ticker needs
   tighter gates or wider gates.

Restructured 2026-04-25 from "characterize gates first, infer outcomes"
to **"backtest exits first, infer gate quality from win rates"** — the
trader-facing direction.

## Why now

After 2026-04-25 we have:

- **15,886 backfill alerts** across 13 tickers and 10 days, generated by
  the production `detectAnomalies()` function on real CSV data
- **740,120 strike_iv_snapshots rows** — full per-minute (spot,
  mid_price, iv_mid) time series for every backfill compound key. This
  is what the backtest reads forward in time from each alert.
- **Three exit-signal types** in production (`iv_regression`,
  `ask_mid_compression`, `bid_side_surge`) — comparison baseline for
  what "right exit" looks like in this dataset

10 days is thin — treat results as directional, not statistically
conclusive. Live data thickens the population over time.

## The 8 core questions

Stratification questions deferred to Phase C, ordered by expected
actionability:

1. **Per-ticker peak-return distribution.** TSLA's 1015 alerts vs
   META's 286 — same density per-key or different?
2. **Signal-stack stratification.** Does `skew_delta + z_score +
ask_mid_div` outperform `skew_delta` only? Does multi-signal predict
   higher peak return?
3. **Time-of-day effect.** Open auction (8:30-9:30 CT) vs institutional
   window (9:30-12:00) vs afternoon (12:00-15:00)?
4. **Vol/OI threshold tuning.** Sweep 5-10× / 10-50× / 50-200× / 200×+
   buckets — does win rate plateau or keep rising?
5. **OTM-distance effect.** For cash-index whales (now captured at ±12%),
   monotonic or sweet-spot?
6. **Side-skew sensitivity.** Does signal quality plateau at 0.65 or
   keep climbing toward 1.0?
7. **flow_phase outcome differentiation.** Does early outperform mid
   outperform reactive?
8. **side_dominant effect.** Do puts on the bid behave differently than
   puts on the ask in outcome distribution?

## Phases (ordered)

### Phase A — Outcome computation (~1h)

Foundation for B and C. Extract per-alert outcome features from
`strike_iv_snapshots` joined forward in time.

**Script:** `ml/extract-iv-anomaly-outcomes.py`

For each backfill anomaly, compute (joining `strike_iv_snapshots` where
`(ticker, strike, side, expiry)` matches and `ts >= alert.ts <= expiry
close + buffer`):

| Outcome feature      | Definition                                                                                       |
| -------------------- | ------------------------------------------------------------------------------------------------ |
| `entry_premium`      | `mid_price` at the first snapshot at/after `alert.ts`                                            |
| `entry_spot_actual`  | `spot` at that same snapshot (vs alert's `spot_at_detect`)                                       |
| `peak_premium`       | `max(mid_price)` between alert and EOD/expiry close                                              |
| `peak_premium_pct`   | `(peak - entry) / entry`                                                                         |
| `time_to_peak_min`   | Minute index where peak occurred                                                                 |
| `first_itm_ts`       | First minute where spot crossed strike (call: spot ≥ strike, put: spot ≤ strike). NULL if never. |
| `first_itm_premium`  | `mid_price` at `first_itm_ts` (NULL if never ITM)                                                |
| `time_to_itm_min`    | Minute index where first ITM hit (NULL if never)                                                 |
| `close_premium`      | `mid_price` at last sample (≥ session close or nearest available)                                |
| `close_pnl_pct`      | `(close - entry) / entry`                                                                        |
| `finished_itm`       | Boolean — was spot ITM at last sample                                                            |
| `intrinsic_at_close` | For 0DTE: `max(spot - strike, 0)` (calls) or `max(strike - spot, 0)` (puts)                      |
| `sample_count`       | How many snapshots were available (data-quality flag)                                            |

Output: `ml/data/iv-anomaly-outcomes.parquet` (~16K rows).

### Phase B — Backtest (~2h)

The trader-facing piece. Three exit playbooks compared per ticker /
per-DTE.

**Script:** `ml/backtest-iv-anomalies.py`

| Strategy              | Exit rule                                | PnL =                                                                                                                  |
| --------------------- | ---------------------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| **Sell-on-ITM-touch** | Sell at first minute spot crosses strike | `first_itm_premium` (or `entry_premium` if never ITM and held to EOD where it expired worthless = 0DTE-only edge case) |
| **Sell-at-peak**      | Cheating oracle (best-case ceiling)      | `peak_premium`                                                                                                         |
| **Hold-to-EOD**       | Hold until 15:00 CT (or expiry close)    | `close_premium` for non-0DTE; `intrinsic_at_close` for 0DTE (worthless if OTM, intrinsic if ITM)                       |

**Per-ticker / per-DTE-bucket aggregates:**

- Win rate (% with positive PnL after entry premium)
- Mean PnL %
- Median PnL %
- p25 / p75 PnL %
- Max loss (worst trade)
- Mean time-to-ITM (for trades that hit ITM)
- % that touched ITM
- % that finished ITM

**DTE buckets:** 0DTE / 1-7DTE / 8-14DTE — different playbooks apply.

**Outputs:**

- `ml/data/iv-anomaly-backtest-2026-04-25.parquet` — per-alert backtest
  outcomes
- `ml/reports/iv-anomaly-backtest-2026-04-25.md` — human-readable per-
  ticker tables + interpretation
- `ml/plots/iv-anomaly-backtest/`:
  - per-ticker strategy comparison bar chart (3 strategies × win rate)
  - time-to-ITM histogram per ticker
  - PnL distribution box plot per strategy
  - 0DTE-specific: % touched ITM but finished OTM (the trap)

**Decision artifact:** for each ticker, "best playbook" recommendation
based on Sharpe-like (mean / stddev) of each strategy's PnL %.

### Phase C — Stratified characterization (~1h)

Now that backtest tells us which alerts are actionable, characterize
the gates that produced them.

**Script:** `ml/stratify-iv-anomalies.py`

For each of the 8 questions, slice the backtest output by that gate
dimension and compute:

- n alerts in each bucket
- win rate per bucket (using best-playbook PnL)
- mean PnL per bucket

Output: `ml/findings/iv-anomaly-stratified-2026-04-25.json` (machine-
readable) + `ml/reports/iv-anomaly-stratified-2026-04-25.md` summarizing
which gate adjustments would improve win rate.

**Key feedback loop:** if a ticker has low win rate AND a particular
gate value (e.g., vol/OI < 10× concentrates losers), the gate tier for
that ticker should tighten. This loops back into production constants.

## Constraints

- **Sample size** — 10 days × 13 tickers; per-ticker subsets get small.
  Treat as directional.
- **0DTE specificity** — finishing OTM on a 0DTE = full premium loss.
  Hold-to-EOD will look terrible for 0DTE alerts that touch ITM but
  retrace. This is the central question, not a flaw.
- **Snapshot density** — backfill `strike_iv_snapshots` are 1-min
  granular per the production cron schema. Outcome computation reads
  the post-alert window from those rows directly. Some strikes may have
  gaps (cron failures, market holidays inside the window) — handle as
  NULL not as "false negative."
- **Slippage / fills not modeled** — backtest assumes execution at
  `mid_price`. Real fills are worse. The relative comparison between
  strategies is meaningful; absolute returns are optimistic.
- **Cross-asset context absent** — `vix_level` only. Phase C macro-
  regime questions (Q3/Q7) work with what's there; deeper macro context
  is a separate enrichment.

## Out of scope

- **Predictive ML model training** — this is exploratory + descriptive,
  not predictive. After 4-6 weeks of live data, a Phase 5 spec can
  revisit modeling.
- **Cross-asset feature engineering** — separate spec to retroactively
  enrich `context_snapshot` from `futures_bars`, `spx_candles_1m`, etc.
- **Strategy optimization** — this study identifies the best of 3
  predefined exit playbooks; designing optimal exits (e.g.,
  trailing-stop, partial-scale-out) is its own feature.
- **Slippage modeling** — pure mid-price assumption is good enough for
  ranking strategies; tuning for $0.05/contract slippage is a v2.
