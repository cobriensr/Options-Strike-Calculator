# UI/UX Audit — 0DTE SPX Strike Calculator

**Date:** 2026-03-29
**Scope:** Full `src/` frontend analysis — visual design, interaction patterns, accessibility, state management UX

---

## Executive Summary

The app is a hand-crafted, data-dense trading dashboard with a strong foundation: CSS-variable theming, warm neutral palette, three-font typography hierarchy, solid accessibility basics, and a clean component architecture. The primary improvement opportunities fall into three buckets:

1. **State communication** — loading, empty, and error states need visual treatment
2. **Visual hierarchy** — sections need tiered visual weight so the eye knows where to focus
3. **Consistency** — border radii, spacing tokens, font sizes, and interactive states have drift

---

## Current Design System

### Color Architecture

**Method:** CSS custom properties in `index.css` with `.dark` class override. Single `theme` object in `src/themes/index.ts` references `var(--color-*)`. No prop threading — dark mode is automatic.

**Light Palette:**
| Token | Hex | Usage |
|-------|-----|-------|
| `--color-page` | `#f4f1eb` | Page background (warm linen) |
| `--color-surface` | `#ffffff` | Card/section backgrounds |
| `--color-surface-alt` | `#edeae3` | Alternate rows, secondary surfaces |
| `--color-primary` | `#1c1a15` | Body text |
| `--color-secondary` | `#4a4740` | Secondary text |
| `--color-tertiary` | `#5c5950` | Labels, tertiary text |
| `--color-accent` | `#1d4ed8` | Interactive elements, highlights |
| `--color-success` | `#15803d` | Positive values, call side |
| `--color-danger` | `#b91c1c` | Negative values, put side |
| `--color-caution` | `#946a00` | Warnings, moderate risk |
| `--color-backtest` | `#7c3aed` | Backtest mode indicator |

**Dark Palette:** Shifts to cooler tones (`#1a1a22` page, `#242430` surface, `#7ba4ff` accent). Radial gradient on `#app-shell` adds subtle depth.

**44 total tokens** covering backgrounds, text hierarchy, borders, tooltips, chips, charts, and selection colors.

### Typography

**Three font families:**
- **Source Serif 4** (default body) — editorial warmth, high readability for data context
- **DM Mono** (data values) — clear numeric disambiguation, tabular alignment
- **Outfit** (labels, UI chrome) — geometric sans for small uppercase labels

**Type scale (pixel-precise):**
| Token | Size | Usage |
|-------|------|-------|
| `--text-2xs` | 8px | Micro annotations |
| `--text-xs` | 10px | Small labels, badges |
| `--text-sm` | 11px | Section labels, chip text |
| `--text-md` | 13px | Body text, table cells |
| `--text-lg` | 15px | Subtitle, descriptions |
| `--text-xl` | 16px | Large body text |

**Tracking scale:** `0.06em` (tight) → `0.08em` (normal) → `0.12em` (wide) → `0.14em` (wider)

**Font size adjust:** CLS prevention via `font-size-adjust` (0.48 serif, 0.52 mono, 0.5 sans).

### Spacing

Currently ad-hoc. Common values found in codebase:
- `gap-2` (8px), `gap-2.5` (10px), `gap-3` (12px), `gap-4` (16px)
- Section margins: `mt-6` (24px)
- Internal padding: `p-[18px]`, `p-3`, `p-4`, `p-[7px_8px]`, `p-[11px_14px]`

### Borders & Shadows

**Border tokens:**
- `--color-edge` (light: `#ddd8ce`, dark: `#383848`) — standard borders
- `--color-edge-strong` — input borders
- `--color-edge-heavy` — heavy emphasis

**Border widths:** `1px` (default), `1.5px` (sections/inputs), `2px` (results), `3px` (accent top bar)

**Shadow scale:**
- Subtle: `0 1px 4px rgba(0,0,0,0.03)` — sections
- Card: `0 2px 8px rgba(0,0,0,0.06)` — elevated
- Tooltip: `0 4px 24px rgba(0,0,0,0.25)` — floating

### Animation

- `animate-fade-in-up`: 250ms ease-out, opacity 0→1 + translateY 6px→0
- `transition-[width] duration-200`: bar width changes
- `transition-all duration-[250ms]`: app shell theme switch
- `transition-colors duration-100`: button hover
- `prefers-reduced-motion: reduce` respected globally

## Findings

### Strengths

1. **CSS variable theming** — elegant, zero-JS dark mode, single source of truth
2. **Warm neutral palette** — distinctive, not another cold blue dashboard
3. **Typography trifecta** — serif/mono/sans serves data-dense content well
4. **`tint()` utility** — consistent progressive color layering throughout
5. **Accessibility foundations** — ARIA labels, focus rings, skip link, semantic HTML, `prefers-reduced-motion`
6. **Lazy loading** — code-split heavy sections with Suspense boundaries
7. **Debounced inputs** — 250ms prevents excessive recalculations
8. **Smart defaults** — time initializes to 10:00 AM CT outside market hours

### Issues Found

#### Critical (High Impact, Low Effort)

| # | Issue | Location | Description |
|---|-------|----------|-------------|
| 1 | No empty state for results | `ResultsSection.tsx` | When `results === null` and no validation errors, nothing renders — no guidance |
| 2 | Identical loading fallbacks | `App.tsx:580-640` | All Suspense fallbacks are `"Loading..."` — no content shape hints |
| 3 | No table row hover | `DeltaStrikesTable.tsx`, all table components | 14-column tables with no row hover — eye loses its place |
| 4 | Emoji for dark mode toggle | `App.tsx:483` | Emoji `☀️`/`🌙` renders inconsistently across platforms |

#### Medium (Consistency & Polish)

| # | Issue | Location | Description |
|---|-------|----------|-------------|
| 5 | Border radius drift | `ui.tsx` (14px), tables (10px), inputs (lg/8px) | Three competing radius values, some as arbitrary `[Npx]` |
| 6 | Typography size inconsistency | Labels across components | Labels use `text-[10px]`, `text-xs`, `text-[11px]` interchangeably |
| 7 | Spacing arbitrariness | Global | `p-[18px]`, `p-[7px_8px]`, `p-[11px_14px]` alongside standard tokens |
| 8 | Hardcoded color | `ThetaDecayChart.tsx` | `#f59e0b` instead of `var(--color-chart-amber)` |
| 9 | Limited transitions | Global | Only `animate-fade-in-up` and bar width transitions exist |
| 10 | Missing disabled states | Buttons across app | No `disabled:opacity-50 disabled:cursor-not-allowed` pattern |

#### Low (Enhancement Opportunities)

| # | Issue | Location | Description |
|---|-------|----------|-------------|
| 11 | Visual hierarchy flat | `App.tsx` layout | All SectionBox components have equal visual weight |
| 12 | No expand/collapse animation | `MarketRegimeSection.tsx` | Show/Hide hard-cuts content |

## Proposed Changes

### Phase 1: Quick Wins (1-2 sessions)

1. **Add empty state to ResultsSection** — muted placeholder: "Enter spot price and IV to calculate strikes"
2. **Add shaped skeleton loaders** for Suspense fallbacks (2-3 pulsing bars matching section shape)
3. **Add `hover:bg-surface-alt` to all table `<tr>` elements**
4. **Replace emoji toggle with SVG sun/moon icons** (Lucide or inline SVG, colored via `currentColor`)
5. **Standardize border radius** — `rounded-xl` (12px) for containers, `rounded-lg` (8px) for inner elements
6. **Replace `#f59e0b` with `var(--color-chart-amber)`** in ThetaDecayChart
7. **Add `disabled:opacity-50 disabled:cursor-not-allowed`** to button patterns

### Phase 2: Consistency Pass (2-3 sessions)

8. **Normalize typography** — standardize labels to `text-xs` (10px), body to `text-md` (13px), use CSS custom property tokens instead of arbitrary `text-[Npx]`
9. **Tighten spacing** — replace arbitrary pixel padding with closest Tailwind tokens
10. **Add `transition-colors duration-150`** to table cells for live-updating value changes
11. **Add expand/collapse animation** to Market Regime toggle (CSS grid-rows or max-height)
12. **Increase chip touch targets** — `py-2` minimum for mobile compliance

### Phase 3: Structural Improvements (3+ sessions)

13. **Tier section visual weight** — inputs (quiet), analysis (medium), results (loud). Different background treatments or border emphasis per tier.
14. **Move VIX CSV upload** from header to IV Input section
15. **Add collapsible input sections on mobile** — summary bar after inputs are set
16. **Add sticky inner headers** for long sections on mobile scroll

## State Management UX Summary

### Loading States (Current)
- **Market data:** Background fetch, status badge shows last update. No spinner — data arrives invisibly. Good.
- **History data:** `loading: boolean` with StatusBadge("Loading..."). Adequate.
- **VIX data:** Instant from localStorage/static JSON. Transparent fallback to API. Excellent.
- **Analysis:** Elapsed timer + thinking messages + audio cue on completion. Best-in-class.
- **Suspense:** Generic `"Loading..."` text. Needs improvement.

### Error States (Current)
- **Calculation:** Inline field errors near inputs via `ErrorMsg`. Good.
- **Network:** Silent failure with fallback (missing VIX1D → VIX × 1.15). Good for single-user.
- **Auth:** StatusBadge("Re-authenticate") with link. Adequate.
- **Analysis:** Retry with backoff (3 attempts), user-facing error message. Good.

### Missing States
- **Empty results** (no inputs yet)
- **Empty market regime** (not authenticated)
- **Section-specific skeletons** (lazy-loaded components)
- **Value change transitions** (computed data updating)

### Mobile (375px) — Light Mode

- Single-column stack works: sections collapse to full-width gracefully
- Chips are legible but **touch targets are tight** — AM/PM chips at ~36px height (should be 44px+)
- Header wraps awkwardly: "Strike Calculator" title, VIX badge, and dark mode button compete for space on one line. Title breaks to two lines ("Strike / Calculator")
- **Issue:** Tables require horizontal scroll on mobile (ScrollHint handles this), but the fade gradient is subtle — easy to miss that there's more content to the right
- **Issue:** On the regime thresholds table, only 4 of 7 columns are visible — user has to scroll horizontally with no visible indicator of how many columns exist
- **Positive:** The VIX term structure cards stack vertically and remain fully readable
- **Positive:** Input fields remain properly sized and tappable

### Cross-Cutting Visual Observations

1. **Visual monotony:** Every SectionBox has identical visual weight (same border, same shadow, same accent bar). The results table — the entire purpose of the tool — visually blends in with the input sections. It needs to be visually "louder."

2. **Color palette strength:** The warm neutral palette is genuinely distinctive in the trading tool space. Most competitors use cold blue/gray/black. The Source Serif 4 body font adds editorial warmth that makes dense data feel less sterile.

3. **Dark mode is the stronger theme:** The dark mode uses cooler tones (#1a1a22 base) with a subtle radial gradient from the accent color — this creates depth that the flat light mode lacks. The light mode could benefit from a similar subtle treatment.

4. **Data density is well-handled:** Despite 14-column tables and dense metric cards, nothing feels cramped. The monospaced font at 13px with consistent padding keeps things readable. The choice to use serif for body and mono for data creates a natural visual separation.

5. **Transition polish is minimal:** Theme switch is smooth, but section open/close, data loading, and value updates all happen instantly with no interpolation. Adding even simple `transition-colors` on table cells and `max-height` on collapsible sections would add noticeable polish.

## SPA-to-MPA Conversion Guidance

### Current Architecture

The app is a single-page React 19 application (Vite + React, no router) where `App.tsx` orchestrates all sections in one component tree. State is managed via ~11 custom hooks at the top level. The Vercel backend (`api/`) is already cleanly separated as serverless functions.

### Recommended Approach: Next.js App Router Migration

The natural migration path is **Next.js App Router** with file-system routing. This aligns with the existing Vercel deployment and provides:

- **Route-based code splitting** (automatic, replaces manual `lazy()` calls)
- **Server Components** for data-heavy sections (regime tables, history)
- **Shared layouts** for header/theme without re-mounting on navigation
- **URL-based state** for backtesting dates, positions, analysis history

### Proposed Route Structure

```
app/
├── layout.tsx              # Shared header, theme provider, dark mode
├── page.tsx                # Calculator (current main view — inputs + results)
├── regime/
│   └── page.tsx            # Market Regime section (VIX analysis, tables, guides)
├── positions/
│   └── page.tsx            # Position Monitor (account overview, trade log)
├── analysis/
│   ├── page.tsx            # Chart Analysis upload + history list
│   └── [id]/page.tsx       # Single analysis detail view
├── risk/
│   └── page.tsx            # Risk Calculator (standalone tool)
├── journal/
│   └── page.tsx            # Trade journal (if adding journal UI)
└── backtest/
    └── page.tsx            # Backtesting mode with date picker + results
```

### Migration Strategy (Incremental)

**Phase 1: Add Router to Existing SPA (minimal disruption)**
- Install `react-router-dom` (or TanStack Router for type-safe routes)
- Wrap the existing app in a `<BrowserRouter>`
- Move each lazy-loaded section to its own route
- Keep all hooks at layout level — route components consume context
- This is a **reversible** step that doesn't require Next.js

**Phase 2: Extract Shared Layout**
- Move header, theme toggle, status badges into a shared `Layout` component
- The calculator inputs become the `/` route
- Market Regime becomes `/regime`
- Position Monitor becomes `/positions`
- Results remain on `/` below the inputs (or become `/results` if separated)

**Phase 3: Next.js Migration (when ready for SSR)**
- Move to Next.js App Router with the route structure above
- Convert data-fetching hooks to Server Components where possible:
  - `useMarketData` → server action that fetches on the server
  - `useHistoryData` → route param `?date=2026-03-30` with server fetch
  - `useCalculation` → stays client-side (pure computation, needs interactivity)
- Add `proxy.ts` (Next.js 16) for auth middleware

### What Stays Client-Side

These hooks **must** remain client-side because they involve interactivity or real-time state:
- `useAppState` (all input state)
- `useCalculation` (pure computation reacting to inputs)
- `useChainData` (polling, WebSocket-adjacent)
- `useImageUpload` (file handling)
- `useAutoFill` (ref-based intent tracking)

### What Can Move Server-Side

These are good candidates for Server Components or server actions:
- VIX range analysis tables (static data, render once)
- Historical regime statistics (9,102 days of data — server render)
- Analysis history list (DB query → render)
- Position monitor initial load (DB query → render)

### URL State Opportunities

Moving to routes enables URL-shareable state:
- `/backtest?date=2026-03-28&time=10:30&tz=ET` — shareable backtest configuration
- `/analysis/abc123` — direct link to a saved analysis
- `/regime?vix=19` — deep link to regime analysis at specific VIX level

### Key Risks

1. **Hook composition disruption:** The 11 hooks in App.tsx share state. Breaking into routes means either lifting state to a context provider or passing via URL params.
2. **Market data polling:** Currently one `useMarketData` instance polls for the whole app. In MPA, you'd need a shared polling layer (context or service worker) to avoid duplicate fetches across routes.
3. **Calculation coupling:** `useCalculation` depends on debounced values from `useAppState` and signals from `useComputedSignals`. These need to stay colocated or communicate via context.

### Recommended First Step

**Phase 1 (React Router) is the lowest-risk path.** It gives you multi-page navigation and URL-based state without changing the build system, deployment, or server architecture. You can evaluate whether Next.js is worth the migration cost after living with routes for a while.
